# v0.5.0 Proof-of-Value Release Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Turn Mnemos from a trustworthy local memory layer into a product that can prove, from real Claude Code sessions, when memory helped, when it missed, and how retrieval quality is changing over time.

**Architecture:** Keep the SQLite-first runtime and Claude Code-first product path. Add a lightweight feedback/event layer for retrieval outcomes, expose that through CLI and MCP, build a review/export surface on top of it, and use captured events to grow a real-world eval pack that can catch regressions better than synthetic benchmark fixtures alone.

**Tech Stack:** Python CLI + MCP server, SQLiteStore, inspectability layer, Claude Code workflows, benchmark harness, Markdown docs.

---

## Release framing

`0.4.0` made Mnemos trustable.

`0.5.0` should make Mnemos provably useful:

- real session feedback instead of only synthetic benchmark results
- explicit tracking of useful recalls, bad recalls, and missed recalls
- a review surface for triaging retrieval quality
- curator behavior tuned by real usage, not just intuition
- release notes that can point to observed usefulness, not just architecture

This is not the milestone for broad host parity or hosted sync.

## Definition of done

Call `0.5.0` ready when all of the following are true:

- a user can record whether a recall was useful, wrong, or missing
- feedback events persist locally and can be audited
- a maintainer can review bad recalls and missed recalls without digging through raw logs
- a real-world recall eval pack can be generated from captured events
- curator heuristics are tighter on noisy facts and validated with tests
- release docs can quote concrete feedback/eval signals instead of only synthetic benchmarks

## Release checklist

### 1. Feedback capture exists

- [ ] retrieval feedback model exists for `helpful`, `not_helpful`, and `missed_memory`
- [ ] feedback events persist in SQLite with timestamps, scope, and source memory IDs where available
- [ ] CLI can record feedback without editing the database manually
- [ ] MCP can record feedback for host-integrated workflows
- [ ] docs explain the feedback loop in plain language

**Ship gate:** a real user can tell Mnemos "that helped" or "you missed this" and the system keeps it.

### 2. Review surface exists

- [ ] feedback events can be listed and filtered
- [ ] a maintainer-facing review command can show the highest-signal misses and bad recalls
- [ ] review output includes scope, query, retrieved chunk IDs, and event type
- [ ] export path exists for external analysis
- [ ] at least one sample review workflow is documented

**Ship gate:** a maintainer can inspect retrieval failures without spelunking through ad hoc logs.

### 3. Real-world eval growth exists

- [ ] feedback events can be transformed into an eval-friendly dataset
- [ ] export format is stable enough to check into `benchmarks/datasets/`
- [ ] at least one seed real-world eval pack exists
- [ ] benchmark docs explain synthetic vs real-world packs
- [ ] CI can run at least one real-world eval check once the pack exists

**Ship gate:** Mnemos can learn from production-like usage instead of only handcrafted scenarios.

### 4. Curator quality improves

- [ ] curator has clearer durable-vs-noisy heuristics
- [ ] obvious ephemeral/tool-noise cases are rejected more often
- [ ] tests cover at least a few "should store" and "should skip" curator cases
- [ ] docs explain what curator is expected to save

**Ship gate:** curator output feels selective instead of eager.

### 5. Release proof is shippable

- [ ] release notes include real usefulness/review signals
- [ ] one public artifact explains the feedback loop and what it revealed
- [ ] Claude Code positioning remains the primary story
- [ ] synthetic benchmark claims stay honest and secondary to real-session evidence

**Ship gate:** the `0.5.0` story is "we measured real usefulness," not "we added more internals."

## Things explicitly out of scope for `0.5.0`

Do not let these delay the release:

- new storage backends

- hosted analytics
- visual dashboard work before CLI review flows exist

## Concrete acceptance checks

Run these before tagging:

- `pytest -q`
- `black --check .`
- `mypy .`
- `python -m mnemos.cli doctor`
- `python -m mnemos.benchmark --stores memory,sqlite --retrievers baseline,engine --dataset-pack claim-driving --top-k 1 --enforce-production-gate`

Then verify manually:

- record one `helpful` recall event
- record one `missed_memory` event
- list/export feedback events
- generate one feedback-driven eval artifact
- confirm curator skips an obvious transient/noisy memory case

## Blunt release test

If a skeptical Claude Code user asks:

> "How do you know Mnemos is actually helping?"

`0.5.0` is ready only if the answer is short, concrete, and measurable:

- we capture recall usefulness explicitly
- we review misses and bad recalls directly
- we grow eval coverage from real usage
- we can show how curator and retrieval improved because of that
