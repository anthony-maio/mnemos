# v0.4.0 User Trust Release Checklist

> Goal: make Mnemos feel reliable, understandable, and worth recommending in real Claude Code use.

## Release framing

`0.3.1` was the platform simplification release.

`0.4.0` should be the trust release:
- one obvious install path
- one obvious memory story
- one obvious explanation for what was stored and why
- fewer host/setup surprises
- benchmark and daily-use evidence strong enough to quote publicly

This is not the milestone for more backend work. The backend decision is done.

## Definition of done

Call `0.4.0` ready when all of the following are true:

- Claude Code daily-use flow works for a fresh user without manual debugging
- the recall + curator workflow is useful enough to feel like product, not plumbing
- inspectability explains stored memories and retrieval behavior clearly
- the SQLite benchmark story is repeatable enough to cite in release notes
- the docs and website tell the same product story in under 30 seconds

## Release checklist

### 1. Claude Code trust path

- [ ] Fresh-machine Claude Code setup works from docs without manual config surgery
- [ ] Plugin or direct MCP path starts cleanly on first launch
- [ ] `mnemos-cli doctor` is green on the default SQLite path
- [ ] hook capture does not fail on stale config or old backend assumptions
- [ ] shared config breakages produce an actionable error, not a dead MCP handshake

**Ship gate:** a new Claude Code user can get to working memory without asking for support.

### 2. Memory workflow feels real

- [ ] `mnemos-recall` workflow is documented and easy to invoke
- [ ] `mnemos-curator` workflow reliably stores durable facts instead of noise
- [ ] end-of-session consolidation is predictable
- [ ] memory scope behavior is obvious in project/workspace/global flows
- [ ] at least 3 realistic Claude Code scenarios are documented and tested:
  - same-repo continuity across days
  - bugfix/history recall
  - cross-project scope isolation

**Ship gate:** a user can feel the product helping with continuity, not just see that memory exists.

### 3. Inspectability and debugging

- [ ] stored memories can be listed and inspected without ambiguity
- [ ] the inspector clearly shows scope, provenance, salience, and graph neighbors
- [ ] retrieval debugging can answer "why did this memory come back?"
- [ ] common failure modes have explicit guidance:
  - empty store
  - wrong scope
  - startup/config mismatch
  - missing embeddings/provider auth

**Ship gate:** when Mnemos behaves unexpectedly, the user can usually diagnose it from the product surface.

### 4. Benchmark credibility

- [x] claim-driving benchmark passes repeatedly on the shipped SQLite path
- [x] benchmark gate is stable enough not to flap on ordinary reruns
- [x] benchmark summary is short enough to include in release notes
- [x] benchmark docs match the enforced gate logic exactly
- [x] at least one benchmark command is copy-paste safe for contributors

**Ship gate:** we can quote the benchmark without apologizing for it.

### 5. Distribution assets

- [ ] release notes explain Mnemos in one clean paragraph
- [ ] one demo artifact exists:
  - short screen recording, or
  - tight side-by-side workflow writeup
- [ ] website messaging matches the SQLite-first trust story
- [ ] Claude Code positioning is sharper than "we also have memory"
- [ ] Codex/Cursor language stays honest: supported, but not hook-parity

**Ship gate:** a technical early adopter can understand why Mnemos matters in under 30 seconds.

## Fastest path to finish

Do these in order:

1. Harden Claude Code startup and config failure handling
2. Productize recall + curator flows for Claude Code
3. Improve inspectability and retrieval debugging
4. Run repeated SQLite benchmark passes and lock the benchmark summary
5. Cut the release notes/demo asset

## Things explicitly out of scope for 0.4.0

Do not let these delay the release:

- new storage backends
- hosted sync/team memory
- Product Hunt launch
- broad host parity across every MCP client
- deeper graph database experimentation
- visual polish work that does not improve trust or usability

## Concrete acceptance checks

Run these before tagging:

- `pytest -q`
- `black --check .`
- `mypy .`
- `python -m mnemos.cli doctor`
- `python -m mnemos.benchmark --stores memory,sqlite --retrievers baseline,engine --dataset-pack claim-driving --top-k 1 --enforce-production-gate`

Then verify manually:

- Claude Code starts Mnemos cleanly on a fresh session
- one memory can be stored, retrieved, inspected, and consolidated
- one scoped-memory scenario and one cross-project isolation scenario behave correctly

## Blunt release test

If a skeptical Claude Code user asks:

> "Why should I use this instead of built-in Claude memory?"

`0.4.0` is ready only if the answer is short, credible, and demoable:

- better scoped memory for repos and workspaces
- inspectable memory behavior
- stronger continuity across sessions
- local-first storage with one-file setup
